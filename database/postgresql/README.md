# PostgreSQL 阶段 1 迁移脚本（coderx）

本目录存放从 MySQL 迁到 PostgreSQL 时**第一版定稿**的 DDL 与校验资产：表结构、触发器、二级索引、以及只读校验查询。

## 建议执行顺序

1. `001_schema.sql` — 建库后执行：表、主键、唯一约束、外键（**不含** `ON UPDATE CURRENT_TIMESTAMP` 的等价逻辑）。
2. `002_triggers.sql` — 补齐 `update_at` 自动刷新（等价 MySQL 的 `ON UPDATE CURRENT_TIMESTAMP`）。
3. `003_indexes.sql` — 创建二级索引（主键与唯一索引已在 `001` 中定义）。
4. `004_verify.sql` — **只读**检查；可在空库跑一遍（结构检查），导入数据后再跑一遍（行数、孤儿行、序列等）。

使用 `psql` 时示例：

```bash
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -f 001_schema.sql
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -f 002_triggers.sql
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -f 003_indexes.sql
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -f 004_verify.sql
```

## `001_schema.sql`（简述）

- 定义 17 张业务表及与 MySQL 对齐的字段名；时间字段统一为 `timestamptz`；整型主键使用 `GENERATED BY DEFAULT AS IDENTITY`。
- 表名 `user` 在 PostgreSQL 中为保留字，脚本中使用 `public."user"`，后续业务 SQL 也需带双引号或改用别名策略。

## `002_triggers.sql` — 做什么、为什么需要

### 目的

在 PostgreSQL 中**复现 MySQL 列定义里常见的** `ON UPDATE CURRENT_TIMESTAMP`：**任意 `UPDATE` 行时，自动把该行的 `update_at` 更新为当前时间**，应用不必每条语句都手写 `SET update_at = NOW()`。

PostgreSQL 没有与 MySQL 完全相同的「列级 `ON UPDATE CURRENT_TIMESTAMP`」语法，因此用 **`BEFORE UPDATE` 触发器 + 一个公共函数**实现等价行为。

### 实现要点

- 函数 `public.set_update_at()`：在更新前执行，将 `NEW.update_at` 设为 `clock_timestamp()`，并 `RETURN NEW`。
- 为所有带 `update_at` 且需要自动维护的表各建一个触发器（共 17 个），名称形如 `{表名}_set_update_at`，均 `EXECUTE FUNCTION public.set_update_at()`。
- 覆盖表：`article`、`article_collect`、`article_history`、`article_like`、`article_tag`、`avatar`、`collect`、`comment`、`comment_like`、`file`、`image_meta`、`profile`、`report`、`tag`、`"user"`、`user_follow`、`video_meta`。

### 注意

若某条 `UPDATE` 显式设置了 `update_at`，当前实现仍会在写入前由触发器覆盖（与「始终自动刷新」策略一致）。若将来需要「显式指定则不覆盖」，需改函数逻辑。

## `003_indexes.sql` — 做什么

### 目的

在 `001` 已创建主键、唯一约束的前提下，补充**查询与维护路径常用的二级索引**，并与 MySQL 侧常见访问模式对齐（列表按时间排序、按用户/外键查、评论树等）。

### 设计说明

- **主键与唯一约束**产生的索引在 `001_schema.sql` 中已存在；本文件只包含 `CREATE INDEX`。
- 全部使用 `CREATE INDEX IF NOT EXISTS`，便于重复执行或环境对齐。
- 包含两类典型索引：
  - **外键/关联列**：如 `article(user_id)`、`article_collect(collect_id)`、`article_history(article_id)` 等，利于连接与按从表查主表。
  - **业务查询与排序**：如 `article(create_at DESC)`、`article_history(user_id, update_at DESC)`、`comment` 上多组组合索引以支持楼中楼、按文章拉取等。
- **部分索引（Partial index）**：
  - `comment_article_root_create_at_idx`：`WHERE comment_id IS NULL`，针对根评论列表。
  - `file_orphan_cleanup_idx`：`WHERE article_id IS NULL`，配合孤儿文件清理任务按 `file_type`、`create_at` 扫描。

执行完成后，应与 `004_verify.sql` 中「第 4 节」列出的索引名一一对应，用于验收。

## `004_verify.sql` — 做什么

本文件**不修改任何对象**，仅包含 `SELECT`（及读取系统目录的查询），用于建表/导入后的**自检清单**。

### 各节概要

| 节 | 内容 |
|----|------|
| **1** | 期望的 17 张基表是否都存在；汇总「已存在表数量 / 期望数量」。 |
| **2** | 带自增主键的表：`id` 列是否为 `IDENTITY`（与 `001` 中 `GENERATED ... AS IDENTITY` 一致）。 |
| **3** | `002` 中定义的 17 个 `*_set_update_at` 触发器是否都存在。 |
| **4** | `003` 中列出的全部二级索引名是否在 `pg_indexes` 中存在。 |
| **5** | 抽样查看部分表的检查约束、唯一约束、外键定义（`pg_constraint`）。 |
| **6** | 导入后各表**行数**（可与 MySQL 同表 `COUNT(*)` 对比）。 |
| **7** | **外键孤儿检查**：每条外键关系上，从表指向主表不存在的行数应为 **0**。 |
| **8** | **语义一致性**：如评论与父评论/回复是否同文、`image_meta`/`video_meta` 是否指向对应 `file_type`；以及每篇文章是否至多一张封面（`HAVING count(*) > 1` 的结果集应为空）。 |
| **9** | **序列与数据对齐**：各表 `max(id)` 与 `pg_sequences.last_value` 对照，用于导入后 `setval` 校准。 |
| **10** | **时区观感**：对若干表输出 UTC 与上海时区下的 `min/max(create_at|update_at)`，辅助确认导入与会话时区未引入明显偏移。 |

### 如何使用结果

- **空库阶段**：重点看第 1–5 节（结构、触发器、索引、约束）。
- **全量导入后**：逐节人工或通过脚本比对第 6–10 节；行数与 MySQL 一致、孤儿为 0、序列合理、语义检查为 0，再进入应用层联调。
